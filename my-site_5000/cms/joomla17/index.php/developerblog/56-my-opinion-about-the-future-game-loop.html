My opinion about the concept of the game loop in the future

Last Updated on Tuesday, 27 December 2011 20:50 | | Hits: 848

These days, multithreaded programming become a very popular topic since the modern CPU has a trend to be manufactured in the multi-cores fashion.

In the past, CPU has been increased its power through its operation frequency. Higher clock they used, higher and faster performance we can have. If we take a look at a basic game loop, the trend of CPU development in those days is perfectly fitted in the way we, game developers, prefer. Considering the game loop performance could be determined from how fast the CPU can update through the whole process, (update, simulate and render) we can see the game application will be able to run faster regarding to the increased frequency.

It does seem like the application will be scaled automatically as the hardware progress. Since we don't have to plan ahead for the application's scalability, then we could assume that the increased performance can be gained for "FREE"

In multi-cores fashion, we won't get any free extra performance without planning. Single-threaded application will not gain the best performance on the Multi-cores processor. Then, can we just multi-threaded our game?

It's true that multi-threaed game loop model that we are after, however, it's not a guarantee that your game application will always gain the huge performance after multithreading. As we concern, the game application usually contain a lot of objects that "share" the data across the different processes. In multithreaded programming, "Shared data" is one of the most troublesome situation that can reduce your application performance dramatically, because of the data access control which can block the parallel operations.

In conclusion, without well-ahead planning, multithreading programmers can spend time for so long trying to optimize their application into the multithreaded fashion. It sounds painful, but they can ended up have no or very little performance gain because of those above restrictions.

There are quite a few techniques that is popular in order to handle the multithreading problem in games. The techniques such as "Double Buffering", "Lock-free data structure" and "Command Queues or Deferred function calls" have been proved very useful and suitable on the modern situations. However, it's still quite complicated, and sounds unnecessary for the higher level programming.

In my opinion, it will be great if it is possible to leave the game application's scalability handled by the compiler and operating system. Considering we need an information of the status of registers and buffers, in order to optimize the multithreading situation. If these operations can be determined dynamically from the lower-level, the operating system could be used as the great hand to scale our application to be optimal on the different hardware!

Currently, I can see from some hardware manufacturers started to put those semi-automatic scalable application concept on to their drivers, compilers and libraries. Intel has their own package called "TBB" that trying to provide the ability to distribute operations among CPU cores automatically, based on the "Task Stealing" concept by using thread-pools and task-pools.

On the graphic card, the new driver, such as DirectX10, the memory operation that have to done on the GPU will be operated under the command queue in order to reduce the blocking situation. The command queue concept also yield out another benefit where each task can be prioritized.

Ultimately, I hope in the future we can scale the game application to multi-cores processor without modifying the concept of basic game loop so it can stay beautiful and simple as it already did =) . According to the above techniques and technologies, I strongly believe that the application has the possibility to be scaled automatically by the smart compiler and the operating system in the coming age.
