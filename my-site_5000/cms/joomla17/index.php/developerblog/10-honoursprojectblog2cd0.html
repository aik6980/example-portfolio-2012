Profiling tools

Last Updated on Monday, 02 January 2012 09:51 | | Hits: 518

Before we start profiling performance on our application, it's very important that we also understand the tools that we use to perform such the operations. We already had a talk about what and where is the potential bottlenecks could occurs. For a quick reminder, there're two main areas of bottlenecks we need to concern, where either on CPU or GPU.

PerfHUD, GPU Profiler from NVidia

On the GPU side, PerfHUD 6 is a visual debugging and profiling tool that can help you find and fix problems in your graphics rendering pipeline.

It does this by letting you explore and edit data in the graphics pipeline, all while your application is running. PerfHUD also offers automated performance analysis to quickly identify the most expensive draw calls.

Even PerfHUD has been created with the highly concern on a seamless intregration between the profiler and user's application. It still requires a small code change in how you create your Direct3D device in order to enable analysis. When this process have been completed, PerfHUD will let you explore an application’s rendering pipeline and methodology in extreme detail.

Intel VTune Performance Analyser, CPU Profiler

Perhaps the most fundamental part of performance tuning involves identifying the location of performance bottlenecks and determining how to fix them. For this task, the best solution requires access to a profiling technology that does not change the performance of the software, so that the measured performance accurately reflects what the application really does.

Ideally, a tool not only finds performance bottlenecks, it also helps the developer generate ideas about what can be done to address bottlenecks. The most difficult part of performance tuning involves determining how to make those bottlenecks go away.

The VTune™ Performance Analyzer provides all of these capabilities for both Windows* and Linux* on Intel 32-bit and 64-bit processors. This tool uses occasional hardware interrupts to identify performance bottlenecks in a low-intrusive technology called Sampling. With this technology, samples of the CPU’s execution context are taken in real time while using post-execution analysis to identify the locations of CPU bottlenecks. This allows the developer to drill down all the way to the source code for specific bottleneck locations.

Strategies for frame rate improvement

Geometry instancing has been proved itself which is an essential technique for increasing the GPU utilization. However, when the scene is contained the huge number of objects (let say 100k or above), it usually shows the dropping of GPU utilization. This is because, we spend time during a frame updating the objects in the CPU main loop instead of passing jobs to GPU for rendering.

In order to yield the best GPU utilization, we want the GPU to be busy all the time. In our situation, we already did by batching large number of objects before stream into the rendering pipeline.
Even regardless of game-play logics, there's still some render-related expensive operation, for example Character Animation. Here we have listed some possible solution which we're going to explore in order to improve our application frame rate.

  • Using "needToUpdate" flag could help us go through the loop much faster where some objects might not need the logic update. This also refers to the use of object classification. Most of the scene object in game might appear to be "Static" or "Non-AI" type of object such as building, grass and trees. These type of objects will be directly benifitted from Geometry instancing, because they are easily batched together.
  • The scene need some sort of "fast searching" for visibility classification. Using "Quadtree" or "Octree" at this point should help us eliminate the number of occluded objects very fast
  • However, it's still possible that there will be a huge number of visible objects in front of the camera. This is where we could use the "Distance-base Level of detail" to shave some works off from the distanced objects.
  • We should be aware of "searching" and "classifying" that could be another performance hoag. The criteria for bucketing objects into different LoD level is very important
    • At the moment, the distance from the camera is the main criteria.
    • The classification process is most likely the "read-only" process. this task could be possibly done on "Multi threading" if needed.
    • Animation update could be completely suppressed on the distance objects
    • Full Detailed Model will represented LoD0, which is used the most expensive "shader" (normal maps, and per pixel lighting).
    • Pre-processed Progressive mesh is used for LoD1 and 2
    • Different type of imposter is planned to used for distanced object
      • Geo-imposter is the cloud of imposter can be used instead of progressive meshes
      • Image imposter, this type of imposter is the low-detaied 2D texture. Rendering them using "Geometry Shader" as particles might help moving more workload onto the GPU.
According to this, we will carry on our experiments through implementing those above algorithms

Weekly update

Quadtree Implementation have been happily done on XNA. The demo has some interesting functions where I can render the "Bounding Frustum" and perform a collision test with the render grid. The result shows the Quadtree visibility culling efficiency, and why we should use it to speed up the scene query process.

Moreover, the implementation also shows the Quadtree module portability. Together with some Events notification (Delegates in .NET), it could not be dead difficult to integrate it into my DX10 Rendering system

  • Bounding Frustum class need to be created manually on C++
  • Quadtree is a 2-dimension spatial query. Consider using 2D AABB could help us speed up the collision testing
Note:

the most optimal tree structure, for searching and querying, is a balanced tree. This is why self-balancing tree can help speeding up the search and sort process.

Direct3D10 Pitfall

I like DX10 very much personally, because its new rendering pipeline functions and structures have been well-named and organized. However, it's hard as well not to say it still looks like it's an incompleted API, especially when compares to the ready-to-work product like XNA :(

I have found some of the faulty functions while I was working on my project for a last few weeks. I've already tried asking to MSDN forum which I hope i can have some good answers very soon

  • ID3DX10Mesh::CloneMesh(), this function is supposed to inherit its functionality from DX9 CloneMesh(), which we can use it for changing the mesh's Vertex Description. However, it's not working properly and also doesn't give out any error :(
