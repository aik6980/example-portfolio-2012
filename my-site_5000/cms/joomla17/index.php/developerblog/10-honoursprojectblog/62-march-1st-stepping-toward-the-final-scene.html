Note from last week discussion

  • First of all, thanks to Henry and Matt for introducing me to Damian (RARE) and allow me to have a private chat with him :)
  • Using "Double buffering" concept should be able to help me extend my application into multi-threading.
  • Suppress animation and mesh's random select will definitely reduce the CPU load (where the bottleneck is meshes LOD -> instance buffer classification)
  • Usually, 3-4 LOD levels is enough in most situations
  • Real time Impostors can be updated and rendered at the top of the frame, before everything else.
  • Objects will be tested and tell the impostors pool that they need one. If nothing to update, we can skip the impostors update process in order to save some frame time
  • Above strategy shows that impostors preparation can be CPU-bound (because they need a lot of stage-changing)
  • Also have a discussion with Stuart Gray. Last year's student which hos final year project is about "Rendering Crowd". His project is about the objects's diversity rather than performance.
  • He spotted that some cheap rendering tricks can be used to avoid crowd similarity, in order to trick player's perception on scene realism.
  • Using "Texture arrays" and some coloring parameters can increase the object variation.
  • Distanced object animation can be replaced by the simple movement (e.g. Cheering animation -> Move object up and down repeatly)
